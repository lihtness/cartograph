# Cartograph — Scaffold Design

## What the Scaffold Is

A minimal directory structure with machine-readable intent declarations, initialized once and extended as the project grows. It is the answer to "where does this go?" for both the user and the AI assistant.

The scaffold is not a template of files to fill in. It is a set of declared concepts — directories with intent — that the project organizes around. Content fills in naturally. The structure persists.

---

## CARTOGRAPH.md — The LLM Navigation File

`CARTOGRAPH.md` lives at the project root. It is the first thing an AI coding agent should read in any session. It is not documentation — it is a routing map.

**Format:**

```markdown
# CARTOGRAPH

docs/ is organized by concept — each section answers one primary question.
When asked to document something, route to the section whose question it answers.
Do not write or update documentation without being asked.

## Document Map

| Directory | Primary Question | Lifecycle |
|-----------|-----------------|-----------|
| docs/thesis/ | Why does this exist? What is the core bet? | Stable |
| docs/architecture/ | How is it built? Key technical decisions. | Moderate |
| docs/quality/ | How do we know it works? Validation methods. | Moderate |
| docs/product/ | What are we building next? Current roadmap. | Fast |
| docs/ops/ | How do we run it? Deployment, security, infra. | Mixed |
| docs/sales/ | How do we explain and sell it? | Fast |

## Memory

| Location | Format | Purpose |
|----------|--------|---------|
| .claude/projects/<hash>/memory/ | Claude memory files | Session artifacts — bridge to stable docs |

## Conventions

- One canonical source per fact. Cross-reference, do not duplicate.
- Memory files graduate to docs when stable. Mark SUPERSEDED when promoted.
- Roadmap in docs/product/ should match git activity. Run `cartograph reconcile` weekly.
- New directory needed? Add it here with a primary_question before creating files.
```

This file is auto-generated by `cartograph init` and updated by `cartograph add-section`. Users can edit it manually; Cartograph reads it as config, not as output.

## scaffold.toml — Machine-Readable Manifest

`.cartograph/scaffold.toml` is the authoritative machine-readable version of the directory map. `CARTOGRAPH.md` is generated from it. Cartograph reads `scaffold.toml` for routing decisions; humans read `CARTOGRAPH.md`.

**Format:**

```toml
[scaffold]
version = "1"
docs_root = "docs"

[[section]]
name = "thesis"
path = "docs/thesis"
primary_question = "Why does this exist? What is the core bet?"
lifecycle = "stable"          # stable | moderate | fast | mixed
intent = """
Core hypothesis, problem statement, domain thesis, central bets.
Changes here indicate a project pivot — treat with that weight.
"""
required = true

[[section]]
name = "architecture"
path = "docs/architecture"
primary_question = "How is it built? What are the key technical decisions?"
lifecycle = "moderate"
intent = """
System design, component decisions, data flow, technical constraints.
Documents the current architecture, not aspirational state.
"""
required = true

[[section]]
name = "quality"
path = "docs/quality"
primary_question = "How do we know it works? What are the validation methods?"
lifecycle = "moderate"
intent = """
Validation approach, signal quality, confidence measures, known gaps.
"""
required = false

[[section]]
name = "product"
path = "docs/product"
primary_question = "What are we building next? What is the current roadmap?"
lifecycle = "fast"
intent = """
Roadmap, requirements, feature specs, build order.
Should reflect actual git activity — reconcile weekly.
"""
required = true

[[section]]
name = "ops"
path = "docs/ops"
primary_question = "How do we run it? Deployment, security, infra, runbooks."
lifecycle = "mixed"
intent = """
Operational details: deployment, security design, infra, monitoring.
Separate from architecture — ops is how you run it, not how it is designed.
"""
required = false

[[section]]
name = "sales"
path = "docs/sales"
primary_question = "How do we explain and sell it? Positioning, pricing, outreach."
lifecycle = "fast"
intent = """
Go-to-market: positioning, ICP, pricing, outreach scripts, one-pagers.
User-guided — only scaffold if the project has a GTM dimension.
"""
required = false
```

Adding a new section: `cartograph add-section <name> --question "<primary_question>" --lifecycle <stable|moderate|fast|mixed>`

## CLI: Init and Scaffold Commands

```
cartograph init [--template <name>] [--discover]
```

**`--template`** selects a starting scaffold. Built-in templates:

| Template | Sections | When to use |
|---|---|---|
| `software` | thesis, architecture, quality, product, ops | Default: any technical project |
| `product` | thesis, architecture, product, sales, ops | Projects with a GTM dimension |
| `research` | thesis, architecture, quality | Research or internal tools |
| `minimal` | thesis, product | Early-stage, not over-scaffolded |

**`--discover`** scans an existing docs directory, infers section intent from content, and proposes a `scaffold.toml`. User confirms before writing. Useful for adopting Cartograph on an existing project without starting over.

```
cartograph add-section <name> --question "<text>" [--lifecycle stable|moderate|fast|mixed]
```

Adds a new section to `scaffold.toml`, creates the directory, writes an `INDEX.md` with the intent description, and regenerates `CARTOGRAPH.md`.

## Directory Structure After Init

`cartograph init --template=product` produces:

```
<project>/
├── CARTOGRAPH.md                  ← LLM navigation file (generated, editable)
├── CLAUDE.md                      ← routing contract for coding agents (generated, editable)
├── .cartograph/
│   ├── config.toml                ← cartograph config
│   ├── scaffold.toml              ← authoritative section manifest
│   ├── manifest.toml              ← dependency graph (empty, built incrementally)
│   └── state.toml                 ← run state (last_run, resolved flags)
└── docs/
    ├── thesis/
    │   └── INDEX.md               ← primary_question + intent description
    ├── architecture/
    │   └── INDEX.md
    ├── product/
    │   └── INDEX.md
    ├── ops/
    │   └── INDEX.md
    └── sales/
        └── INDEX.md
```

Each `INDEX.md` contains only the section's `primary_question` and `intent` text. No boilerplate, no required fields. Content fills in at the user's pace.

## CLAUDE.md Template — The Routing Contract

`cartograph init` generates (or appends to) `CLAUDE.md` with a documentation structure section. Its job is narrow: tell the agent where things belong so that when the user asks to capture something, it lands in the right place with no ambiguity.

**Generated section:**

```markdown
## Documentation Structure

docs/ is organized by concept. CARTOGRAPH.md at the project root is the routing map.
Each section declares a primary_question — the question that content in that section answers.

When asked to document something, read CARTOGRAPH.md and route to the section
whose primary_question the content answers.

Do not write or update documentation without being asked.
The user decides what gets captured and when.
```

Four sentences. The agent knows the structure exists, knows where to find it, and knows to wait to be asked. No further behavioral prescription.

## How the Structure Evolves

The scaffold is a starting point, not a constraint. Three growth patterns:

**1. Content fills existing sections.** Docs accumulate in the declared directories. No action needed. Cartograph reads them on reconcile.

**2. A new concept emerges that doesn't fit existing sections.** User (or agent, when asked) runs `cartograph add-section`. The new section is declared with intent before files are created. This prevents content drift into the wrong directories.

**3. A section grows large enough to split.** Example: `architecture/` accumulates both design docs and pipeline-stage references. Split into `architecture/design/` and `architecture/pipeline/` by adding subsection entries to `scaffold.toml`. Cartograph supports nested sections.

The system never forces a reorganization. Sections declared in the scaffold but never populated are simply empty — they create no noise.

## LLM Routing in Practice

When an agent starts a session, it reads `CARTOGRAPH.md`. Every routing decision uses the declared section map:

- "Where do I put a new doc about the embedding strategy?" → `architecture/` (How is it built?)
- "Where do I put updated pricing?" → `sales/` (How do we explain and sell it?)
- "Where do I look for what's being built next?" → `product/` (What are we building next?)
- "The user mentioned a thesis-level pivot — what do I update?" → `thesis/`, then check `product/` for downstream impact

The `primary_question` column is the routing key — written specifically to be unambiguous at decision time without needing additional context.

The `lifecycle` field tells the agent how carefully to treat an edit: a change to `thesis/` is flagged as significant; a change to `product/` is routine.
